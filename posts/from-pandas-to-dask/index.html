From Pandas to Dask

In a project that my team and I recently completed in Cazana, we engineered a data processing pipeline to handle hundreds of gigabytes of data. Since the data processing pipeline was going to be part of a larger machine learning pipeline to be built in Python, for the parts of the pipeline involving large amounts of data we decided to use Dask.

Dask is a Python library that extends Numpy and Pandas to parallel, larger-than-memory and distributed computing. Since Dask extends Numpy and Pandas, it operates naturally in combination with these two libraries, which are familiar to people using Python for machine learning. To leverage this familiarity, Dask mimics the API interface of Numpy and Pandas for an easier learning experience.

Despite the easier learning experience, using Dask may prove quite challenging in certain situations because, at its core, it is different from Numpy and Pandas. What I learned when facing some of these challenges make up the content of this article. In particular, this article focuses on the transition to Dask for someone used to Pandas.

Hence, this article is not an introduction or a guide to Dask; for that, you may instead refer to the official Dask documentation. Moreover, this article assumes that you are familiar with Pandas.

Before looking at what I learned, let us start with an overview of Dask DataFrames, which are the analogue of Pandas DataFrames.

Dask DataFrames

A Dask DataFrame is a collection of Pandas DataFrames managed by Dask.

Splitting a Pandas DataFrame into a collection of smaller DataFrames helps in these situations:

  • A Pandas DataFrame is too large to fit in memory (RAM). We can split the Pandas DataFrame into smaller DataFrames, save them to disk and load each one in memory only when needed.
  • Pandas DataFrame operations are using just one CPU core. If there are multiple cores, we can split a Pandas DataFrame into smaller DataFrames, assign each one to a different core and carry out the operations in parallel.
  • We want to leverage the power of a computer cluster, which can be thought of as a system with multiple machines working in parallel, each one with its CPU, memory and disk storage. This case can be thought of as an extension of the above case.

More generally, splitting a Pandas DataFrame into smaller partitions and distributing the DataFrame operations over multiple CPU cores or machines are the building principles of Dask.

Since Dask is built on top of Pandas, it adds a level of complexity. Hence, it is better to use Dask only if you are in at least one of the three situations listed above, otherwise, Pandas is preferred. The great thing about Dask is that you do not have to choose either Dask or Pandas, but can naturally use a mix of both. This means that, in your data processing pipeline, you can turn to Dask only when you need to and then revert to Pandas.

Turning to Dask only when needed is good for the following reasons:

  • Dask is slower than Pandas on data that fits in the memory of an average laptop or desktop computer, as we will see in a benchmark, below.
  • Some Pandas operations are not possible in Dask because of the different way that Dask handles data. For example, aggregations that require the whole data, such as calculating the quantile, are not possible with Dask DataFrames because the data is split into multiple Pandas DataFrames. However, there may be approximate aggregations, like the approximate quantile, that can be used in place of the exact ones.
  • Since Dask is a project much younger than Pandas, some features from Pandas are not yet implemented in Dask. For example, multi-index operations are still limited in Dask. To get a quick idea of what Dask DataFrames may be missing from Pandas DataFrames, you may search for "Not implemented" or "Not supported" in the Dask DataFrame API page.
  • Dask introduces complexity on top of Pandas to solve the three problems introduced at the beginning. Typically, complexity comes at a cost, and Dask is no exception. With more complexity, the user has to care for more things, like the memory size of the Dask DataFrame partitions. Hence, it is better to avoid unnecessary complexity by using Pandas DataFrames where possible.

Now that you have a better idea of the advantages and disadvantages of Dask DataFrames, let us see how to use Dask DataFrames effectively.

Changing the Index

A Dask DataFrame is a collection of Pandas DataFrames, called partitions, together with operations acting on those partitions. Dask uses the index of the data to create the partitions, whose number or memory size is defined by the user.

The most important thing to keep in mind when using Dask DataFrames is that operations that shuffle the index of the Dask DataFrame are computationally expensive because they move data between partitions. Partitions are stored in the disk, making the operations even slower than if they were stored in memory. Hence, avoid shuffling the index whenever possible, or, with more words, do not mess with the index.

If you are using Dask in just a few parts of the data pipeline in place of Pandas, it is easy to limit operations involving the index. However, if you are using Dask in most of your data pipeline, that you are better off architecting the pipeline in a way that reduces the number of index operations.

A common operation that shuffles the index is changing the number of partitions using the repartition method. Changing the number of partitions means that part of the data from one partition has to be moved to other partitions, leading to a data shuffle. However, changing the number of partitions leads to partial data shuffle, meaning that it may not be that bad.

Another common operation on Pandas DataFrames is setting the index. However, in Dask, setting to index leads to a full data shuffle because Dask partitions the data using the index, and changing the index shuffles all the data in all the partitions. Hence, setting the index is a very expensive operation in Dask. However, sometimes it has to be used. So, let us have a more detailed look at the operation of setting the index in Dask.

First, we import the libraries that will be used throughout the article.

import numpy as np
import pandas as pd
import dask.dataframe as dd

Then, we create a Pandas DataFrame (df) with 3 columns, each with 100,000 random values.

df = pd.DataFrame(
    data={
        "a": np.random.permutation(100_000),
        "b": np.random.random(100_000),
        "c": np.random.random(100_000),
    }
)
print(df.head())
OUT:

       a         b         c
0  48680  0.740794  0.486614
1  38283  0.640355  0.003733
2  86664  0.075191  0.997144
3  75611  0.638546  0.056039
4  83474  0.217691  0.971367

Column a is a random permutation of the current index, while columns b and c are random numbers between 0 and 1.

From the Pandas DataFrame, we create a Dask DataFrame (ddf) with 20 partitions.

ddf = dd.from_pandas(df, npartitions=20)
print(ddf)
OUT:

                    a        b        c
npartitions=20
0               int64  float64  float64
5000              ...      ...      ...
...               ...      ...      ...
95000             ...      ...      ...
99999             ...      ...      ...
Dask Name: from_pandas, 20 tasks

The last line specifies that there are 20 tasks. In Dask, a each task is a unit of computation and multiple tasks form a computational graph, which can be thought as a network of tasks. Expressing the tasks using a network is what allows Dask to distribute the computation across different CPUs or machines.

In our case, the Dask DataFrame uses a computational graph made up of 20 tasks, each task corresponding to the creation of each partition. Every time this Dask DataFrame computes something, it run the tasks defined in the computational graph. The input data "flows" through the network to produce the output. This flow of operations is lazy, meaning that Dask re-runs the 20 tasks every time the result of that computation is explicitely required. The computation being lazy can be seen from the absence of values in the columns of the Dask DataFrame printed above, since the print function does not trigger the computation.

As an example of explicitly requiring the above computation, we take the head the Dask DataFrames, which triggers the computation and returns the same result that we had for Pandas, which we then print.

print(ddf.head())
OUT:

       a         b         c
0  48680  0.740794  0.486614
1  38283  0.640355  0.003733
2  86664  0.075191  0.997144
3  75611  0.638546  0.056039
4  83474  0.217691  0.971367

Operations on DataFrames that are small compared to the amount of memory of the machine are slower in Dask than in Pandas. To show this with an example, we print the time it takes to sum columns b and c, first with Pandas and then with Dask.

start_time = time()
df["b"] + df["c"]
end_time = time()
print(f"Pandas DataFrame: {(end_time - start_time) * 1000:7.3f} ms")

start_time = time()
(ddf["b"] + ddf["c"]).compute()
end_time = time()
print(f"Dask DataFrame:   {(end_time - start_time) * 1000:7.3f} ms")
OUT:

Pandas DataFrame:   1.733 ms
Dask DataFrame:    32.211 ms

Dask is about one order of magnitude slower than Pandas. This is due to the overhead of operations in Dask, like creating the computational graph, calling the sum function for each partition and joinoing the result of the partitions into a single output Pandas DataFrame.

Operations that shuffle the data increase the computation gap between Dask and Python even more, as we see in the following example, where we set column a as the new index, first with Pandas and then with Dask.

start_time = time()
df.set_index("a")
end_time = time()
print(f"Pandas DataFrame: {(end_time - start_time) * 1000:7.3f} ms")

start_time = time()
ddf.set_index("a").compute()
end_time = time()
print(f"Dask DataFrame:   {(end_time - start_time) * 1000:7.3f} ms")
OUT:

Set index:
  Pandas DataFrame:   2.492 ms
  Dask DataFrame:   972.854 ms

Dask is now almost two orders of magnitude slower than Pandas.

[^]: For a more accurate comparison, the operation should be run multiple times and averaged. However, here we are only interested in the order of magnitude.

While operations that benefit from parallel processing may have an advantage over Pandas with a large dataset, shuffling the index gets worse with more data.

Setting the Index and Sorting

Unlike Pandas, Dask does not have an explicit method to sort a DataFrame by a column. To achieve a similar result in Dask, we have to set the index of the DataFrame to the column by which we want to sort it. Let us check this with an example and start by creating a Dask DataFrame with a 2 integer columns and 3 string columns split into 2 partitions.

df = pd.DataFrame(
    data={
        "A": np.arange(6),
        "B": np.random.permutation(6),
        "C": np.array(list("abcdef")),
        "D": np.array(list("fbacde")),
        "E": np.array(list("aabaab")),
    }
)

ddf = dd.from_pandas(df, npartitions=2)
print_ddf(ddf)
OUT:

   A  B  C  D  E
---------------- 1st partition
0  0  2  a  f  a
1  1  3  b  b  a
2  2  0  c  a  b
---------------- 2nd partition
3  3  4  d  c  a
4  4  1  e  d  a
5  5  5  f  e  b

Column A goes from 0 to 5, following the index. Column B is a permutation of column A. Column C has the first 6 letters of the alphabet in order. Columns D is a permutation of column C. Column E has a random sample of the first 6 letters of the alphabet. print_ddf is a custom function that prints a Dask DataFrame and show how it is partitioned (TODO: Link to file).

First, let use check the effect of resetting the index of the DataFrame.

ddf = dd.from_pandas(df, npartitions=2).reset_index()
print_ddf(ddf)
OUT:

   index  A  B  C  D  E
----------------------- 1st partition
0      0  0  2  a  f  a
1      1  1  3  b  b  a
2      2  2  0  c  a  b
----------------------- 2nd partition
0      3  3  4  d  c  a
1      4  4  1  e  d  a
2      5  5  5  f  e  b

The partitions remain the same, but the index has been reset independently in each partition. Indeed, the index in each partition restarts from 0. The old index is stored into a colum of the DataFrame called index, like in Pandas.

Let us check what happens if we set column B, which is unsorted, as the index.

ddf = dd.from_pandas(df, npartitions=2).set_index("B")
print_ddf(ddf)
OUT:

   A  C  D  E
B
------------- 1st partition
0  2  c  a  b
1  4  e  d  a
------------- 2nd partition
2  0  a  f  a
3  1  b  b  a
4  3  d  c  a
5  5  f  e  b

The first change that we notice is the different size of the partitions. The first partition has two rows, while the second partition has four. Hence, Dask does not preserve the partition size when setting the index using an unsorted column.

The second change is that the values of column B, that now defines the index, were sorted ascendingly. Indeed, this is the way to sort a Dask DataFrame according to a column. Indeed, unlike a Pandas DataFrame, a Dask DataFrame does not have a sort_values method.

Column C has letters sorted in alphabetical order. Let us see what happens when we set it as the index.

ddf = dd.from_pandas(df, npartitions=2).set_index("C")
print_ddf(ddf)
OUT:

   A  B  D  E
C
------------- 1st partition
a  0  2  f  a
b  1  3  b  a
c  2  0  a  b
------------- 2nd partition
d  3  4  c  a
e  4  1  d  a
f  5  5  e  b

When setting column C as the index, as the letters are already sorted alphabetically, there is no change in the order of the rows.

Column D is a permutation of column C, meaning that it does not have duplicated values, but the values are unsorted. Let us see what happens when we set it as the index.

ddf = dd.from_pandas(df, npartitions=2).set_index("D")
print_ddf(ddf)
OUT:

   A  B  C  E
D
------------- 1st partition
a  2  0  c  b
b  1  3  b  a
------------- 2nd partition
c  3  4  d  a
d  4  1  e  a
e  5  5  f  b
f  0  2  a  a

When setting column D as the index, the rows are sorted alphabetically and the size of the partitions changed, similarly to what happened with column B, whose numerical values are unsorted.

The values of column E are unsorted and duplicated letters. Setting column E as the index leads to the following partitions.

ddf = dd.from_pandas(df, npartitions=2).set_index("E")
print_ddf(ddf)
OUT:

   A  B  C  D
E
------------- 1st partition
------------- 2nd partition
a  3  4  d  c
a  4  1  e  d
a  0  2  a  f
a  1  3  b  b
b  5  5  f  e
b  2  0  c  a

The first partition is now empty and all the rows are in the second partition. Indeed, even if Dask tries to balance the size of partitions, it does not preserve it. This behaviour is is emphasised for very small DataFrames, like this one. For larger DataFrames, the partitions should more or less be even, unless there are too many duplicated values in the index.

However, as can be seen from all the above opertations setting different indices, even if the the partition size may vary, the number of partitions remains the same.

Beside sorting the index, when setting a string column as the index, Dask isolates each duplicated value in a single partition. Indeed, all the rows with index a are in the same partition and similarly for b. This can be seen more clearly if we use a Dask DataFrame with 3 partitions.

ddf = dd.from_pandas(df, npartitions=3).set_index("E")
print_ddf(ddf)
OUT:

   A  B  C  D
E
------------- 1st partition
------------- 2nd partition
a  0  2  a  f
a  1  3  b  b
a  4  1  e  d
a  3  4  d  c
------------- 3rd partition
b  5  5  f  e
b  2  0  c  a

In this case, all the rows with index a are in the second partition and all the rows with index b are in the third partition.

The isolation of rows with the same index to same partition is very useful for certain group-by operations, as we will see next.

Repartitioning with String Index

As we saw in the previous section, setting the index using a string column isolates its distinct values into single partitions. However, unlike setting the index, ripartitioning a string index does not preserve this isolation. Let us see this with an example.

First, we create a Dask DataFrame with a string index and a single partition.

df = pd.DataFrame(
    data={"A": np.arange(6)},
    index=pd.Index(np.array(list("ababab")), name="index"),
)

ddf = dd.from_pandas(df, npartitions=1)
print_ddf(ddf)
OUT:

       A
index
-------- 1st partition
a      0
a      2
a      4
b      1
b      3
b      5

As we have a single partition, the distinct values of the index are isolated in that single partition. If we repartition the DataFrame into 3 partitions, we get the following result.

ddf = dd.from_pandas(df, npartitions=1).repartition(npartitions=3)
print_ddf(ddf)
OUT:

       A
index
-------- 1st partition
a      0
a      2
-------- 2nd partition
a      4
b      1
-------- 3rd partition
b      3
b      5

Value a is present both in the first and in the second partitions and value b is present both in the second and third partitions. So, we lost the isolations of the distinct values into single partitions, and the reason is that the repartition and set_index methods allocate the data into the partitions in different ways. set_index preseves the isolation, while repartition does not. To confirm this, we repartition the data using set_index by resetting the index and setting it again with 3 partitions — note that the reindex method is not implemented in Dask.

ddf = (
    dd.from_pandas(df, npartitions=1)
    .reset_index()
    .set_index("index", npartitions=3)
)
print_ddf(ddf)
OUT:

       A
index
-------- 1st partition
-------- 2nd partition
a      0
a      2
a      4
-------- 3rd partition
b      1
b      3
b      5

Now, all the rows with index value a are only in the second partition and all the rows with index value b are only in the third partition.

In older versions of Dask, specifying the number of partitions in the set_index method may not work. In this case, we can use the following alternative.

ddf = (
    dd.from_pandas(df, npartitions=1)
    .reset_index()
    .repartition(npartitions=3)
    .set_index("index")
)
print_ddf(ddf)
OUT:

       A
index
-------- 1st partition
-------- 2nd partition
a      4
a      0
a      2
-------- 3rd partition
b      1
b      3
b      5

This last way of repartitioning may also be usful in cases when specifying the a partition size in the set_index method does not change the number of partitions.

Aggregation Operations on Groups

Certain aggregation operations, like the median or the quantile, are not implemented for grouped-by DataFrames or Series in Dask. Let us consider the median and show a possible way to calculate it for different groups.

If we have a Dask DataFrame that is too large to fit in memory but columns A and B do fit, to calculate the median, we can simply select the two columns, create a Pandas DataFrame from them using the compute method, and calculate the medians using Pandas.

ddf[["A", "B"]].compute().groupby("A")["B"].median()

However, if columns A and B do not fit in memory, we ca resort to a different method. Let us illustrade this with an example, starting with the following Pandas DataFrame.

df = pd.DataFrame(
    data={
        "A": list("ababab"),
        "B": np.arange(6),
    }
)
print(df)
OUT:

   A  B
0  a  0
1  b  1
2  a  2
3  b  3
4  a  4
5  b  5

Grouping by column A, we get the following medians for column B.

medians = df.groupby("A")["B"].median()
print(medians)
OUT:

   B
A
a  2
b  3

Now, let us create a Dask DataFrame with 2 partitions from the above Pandas DataFrame.

ddf = dd.from_pandas(df, npartitions=2)
print_ddf(ddf)
   A  B
------- 1st partition
0  a  0
1  b  1
2  a  2
------- 2nd partition
3  b  3
4  a  4
5  b  5

In order to calculate the median of each group a and b, we isolate group labels in each partition by setting the index to column A. As we saw in the previous section, this operation puts all the rows with the same value of column A in the same partition.

ddf = ddf.set_index("A")
print_ddf(ddf)
   B
A
---- 1st partition
---- 2nd partition
a  0
a  2
a  4
b  1
b  3
b  5

With the Dask DataFrame in this form, we can apply a group-by operation to each partition using the map_partitions method. This method maps a function to each partition, which is treated as Pandas DataFrame, meaning that we can use Pandas methods. This allows us to use the median method from Pandas, since there is no such method for grouped-by Dask DataFrames. In this way, we get the correct result.

medians = ddf.map_partitions(
    lambda partition_df: partition_df.groupby("A").median()
)
medians = medians.compute()
print(medians)
OUT:

   B
A
a  2
b  3

However, note that this procedure requires a complete shuffling of the index of the Dask DataFrame, which, as we saw above, is very slow.

Disk Cache

In section Changing the Index, we saw how, every time compute method is called on a Dask DataFrame, Dask recomputes all the operations associated to that Dask DataFrame. To see this more clearly, let us create a Dask DataFrame and apply some operations on it.

df = pd.DataFrame(
    data={"A": np.arange(6), "B": np.array(list("xxyzyz"))},
    index=np.array(list("abcdef")),
)
ddf = dd.from_pandas(df, npartitions=2)
ddf = ddf.categorize(columns=["B"], index=True)
ddf["C"] = ddf["A"] ** 2

print_ddf(ddf, end="\n\n")
print(ddf)
   A  B   C
----------- 1st partition
a  0  x  0
b  1  x  1
c  2  y  4
----------- 2nd partition
d  3  z   9
e  4  y  16
f  5  z  25


Dask DataFrame Structure:
                   A                B      C
npartitions=2
a              int64  category[known]  int64
d                ...              ...    ...
f                ...              ...    ...
Dask Name: assign, 10 tasks

The first operation casts column B to a categorical data type. The second operation squares column A and stores the result in a new column, C. Printing the Dask DataFrame after these operations shows that there are 10 tasks associated to it.

If we have to use DataFrame ddf in multiple places, every time the compute method is called, Dask is going to recompute all the 10 tasks defined in ddf because Dask uses lazy computation, meaning that it computes a value only when it is needed, and does not store the output of the computation in memory. Therefore, it is up to us to decide when to persist the result of a Dask computation.

Since Dask DataFrames are typically too large to be stored in memory, the DataFrame that results from a computation may instead be stored in the disk. Dasks suggests to persist DataFrames to disk using the Parquet format because it a format that allows for efficient reading and writing operations. So, let us save the above Dask DataFrame to disk and reload it immediately from there.

[^ ]: Even if the data is stored in a directory, rather than in a file, we use the parquet extension to explicitely show that such directory can be loaded as Parquet data.

data_dir = Path("data.parquet")
ddf.to_parquet(data_dir)
ddf_cached = dd.read_parquet(data_dir)

print_ddf(ddf_cached, end="\n\n")
print(ddf_cached)
OUT:

   A  B   C
----------- 1st partition
a  0  x  0
b  1  x  1
c  2  y  4
----------- 2nd partition
d  3  z   9
e  4  y  16
f  5  z  25


Dask DataFrame Structure:
                   A                  B      C
npartitions=2
               int64  category[unknown]  int64
                 ...                ...    ...
                 ...                ...    ...
Dask Name: read-parquet, 2 tasks

ddf_cached is the same DataFrame that we had before, but has 2 tasks, instead of 10. The 2 tasks correspond to reading the 2 partitions from disk. Hence, every time the compute method is called, the 2 partitions are read from disk. More generally, this means that persisting the data to disk is effective only when loading the partitions from disk is faster than computing the tasks.

To manage the persistance of Dask DataFrames to disk more easily, you may also use the utility class, DaskDiskCache, defined in this file (TODO: link to file). Here is an example usage.

dask_disk_cache = DaskDiskCache()
ddf = dask_disk_cache.persist(ddf, cache_name="data")

print(f"Cache names: {dask_disk_cache.cache_names}", end="\n\n")

print_ddf(ddf, end="\n\n")
print(ddf, end="\n\n")

dask_disk_cache.remove(cache_name="data")
dask_disk_cache.cleanup()
OUT:

Cache names: ['data']

   A  B   C
----------- 1st partition
a  0  x  0
b  1  x  1
c  2  y  4
----------- 2nd partition
d  3  z   9
e  4  y  16
f  5  z  25


Dask DataFrame Structure:
                   A                B      C
npartitions=2
               int64  category[known]  int64
                 ...              ...    ...
                 ...              ...    ...
Dask Name: categorize_block, 4 tasks

Note that, when persisting to disk a Dask DataFrame with categorical columns, the categorical information is not shared across the partitions. So, when loading the data, in orded to synchronise the categorical information across the partitions, we would have to re-categorise the data with the categorize method. This is already implemented in the above utility class, DaskDiskCache.

Debugging

While developing or debugging a data processing pipeline, it is convenient to restrict the data to a small subset of the entire data, so that we can quicly run through the pipeline. However, sometimes, particularly while debugging, we have to use a large subset of the data or perhaps the whole data. In this case, running multiple times the entire pipeline from the start is inconvenient.

As a possible solution, we can create checkpoints throughout the pipeline to persist the data in order to avoid reprocessing it. To do so, we can use the Checkpoints utility class defined in the checkpoints module, which can be found at this link (TODO: link to the module). This class simplifies the creation of checkpoints throughout the pipeline. For example, let us say that we have a function that creates a Dask DataFrame and does some operations on it, and we would like to set a checkpoints so that the function is not re-run every time it is called. To do so, we just have to import the checkpoints module and decorate the function using the checkpoints.allow_checkpoint decorator.

import checkpoints


@checkpoints.allow_checkpoint
def create_dask_dataframe() -> dd.DataFrame:
    df = pd.DataFrame(
        data={"A": np.arange(6), "B": np.array(list("xxyzyz"))},
        index=np.array(list("abcdef")),
    )
    ddf = dd.from_pandas(df, npartitions=2)
    ddf = ddf.categorize(columns=["B"], index=True)
    ddf["C"] = ddf["A"] ** 2

    return ddf

The checkpoints.allow_checkpoint decorator persists the output of the function to disk and loads it from disk if the function is called again.

The above function creates a Dask DataFrame with 10 tasks, like in the previous section. As in the previous section, if we enable the checkpoints using the checkpoints.enable_checkpoints function, every time the function is called, we get 4 tasks, instead of 10.

checkpoints.enable_checkpoints()

ddf = create_dask_dataframe()
print(f"First call -> Number of tasks with checkpoint: {len(ddf.dask)}")

ddf = create_dask_dataframe()
print(f"Second call -> Number of tasks with checkpoint: {len(ddf.dask)}")
First call -> Number of tasks with checkpoint: 4
Second call -> Number of tasks with checkpoint: 4

The first call to the function process the data, stores the output to disk and reloads it. The second call to the function only loads the data from disk.

Since Dask uses lazy evaluation, if the data is not actually used, it is not loaded into memory. This is very useful when combined with the checkpoints decorator defined above. To see why, let us consider the following mock data pipeline.

ddf = create_dask_dataframe()
ddf = engineer_features(ddf)
ddf = preprocess_for_modelling(ddf)
df = ddf.compute()

If the three functions are decorated with the checkpoint decorator, the compute method will load the data from the last function, skipping the first two because the compute method is never called on their outputs.

Load Data from SQL Database

While Pandas allows to read data from an SQL database using a string SQL query through the read_sql_query method, this can not yet be done in Dask. Instead, Dask requires the read_sql_table method and SQLAlchemy. Here is an example.

import sqlalchemy

sql_connection_string = "dialect+driver://username:password@host:port/database"

sql_connection = sqlalchemy.create_engine(sql_connection_string)
metadata = sqlalchemy.MetaData(schema="schema_name")
table = sqlalchemy.Table(
    "table_name", metadata, autoload=True, autoload_with=sql_connection
)

query = (
    sqlalchemy.select(
        [
            table.c.id,  # Type: BIGINT
            table.c.a,  # Type: INT, no nulls
            sqlalchemy.cast(table.c.b, sqlalchemy.Float()).label(
                "c"
            ),  # Type: INT, with nulls
            table.c.c,  # Type: VARCHAR
            table.c.d,  # Type: DATETIME
        ]
    )
    .where(
        sqlalchemy.and_(
            table.c.a > 0,
            table.c.d.isnot(None),  # Drop null values
        )
    )
    .limit(10_000)
    .alias("my_data")  # Any alias name would work
)

dtypes = {
    "id": "int64",
    "a": "int32",
    "b": "float32",
    "c": "object",
    "d": "datetime64[ns]",
}
metadata = (
    pd.DataFrame(columns=dtypes.keys())
    .astype(dtypes)
    .set_index("id", drop=True)
)

# Query
dask_partition_size = 268_435_456  # Bytes == 256 MiB
ddf = dd.read_sql_table(
    table=query,
    uri=sql_connection_string,
    index_col="id",
    bytes_per_chunk=dask_partition_size,
    meta=metadata,
)

First, we define an SQLAlchemy connection string, which we use to create an engine. Using the engine, we create an interface to the SQL table that we wish to query. In turn, this table interface lets us specify the table columns. Next, we define the query using the SQLAlchemy query API. For each Dask DataFrame column that we expect from the query, we define its data type, which is the Pandas analog of the SQL data types of the table. Finally, we specify a partition size and run the query.

Note that we should also specify an index column (id, in our case) that Dask uses to define the DataFrame partitions.

Testing

When using Dask DataFrame in a data processing pipeline, it is important to make sure that the output of the data processed by Dask is independent of the partition size. This is normally the case, but there are exception. For example, if in section Repartitioning with String Index we used the map_partitions method after repartitioning the DataFrame without resetting the index, the result of the aggregation would have been wrong and dependent on the number of partitions.

To make sure that the output of operations over a Dask DataFrame is independent of the number of partitions, we can use a unit test that does so. For example, let us say that we have a function that evaluate the mean squared error (mse) between two columns of a Dask DataFrame.

def mse(ddf, column_a, column_b):
    return ((ddf[column_a] - ddf[column_b]) ** 2).mean()

Then, a unit test for this function would validate the output for a DataFrame with different number of partitions, as shown in the following example.

import unittest


class TestMse(unittest.TestCase):
    def test(self):
        df = pd.DataFrame(
            data={
                "A": [2.0, 5.0, 2.0],
                "B": [1.0, 5.0, 4.0],
            }
        )

        expected_output = 1.6666666666666667

        for npartitions in [1, 2, 3]:
            with self.subTest(npartitions=npartitions):
                ddf = dd.from_pandas(df, npartitions=npartitions)
                output = mse(ddf, "A", "B").compute()
                self.assertEqual(output, expected_output)

In this example, the output is a floating number. Instead, if the output is a Pandas or a Dask DataFrame, we can use the pandas.testing.assert_frame_equal and dask.dataframe.utils.assert_eq methods.

Conclusions

In this article, we saw some challenges that arise when moving from Pandas to Dask and how to deal with them:

  • Operations that shuffle the index of a DataFrame are computationally much more expensive in Dask than in Pandas and should be avoived wherever possible.
  • Sorting the index in Dask is done by setting the column to sort by as the new index, but this involves shuffling the index, which is expensive.
  • Certain aggregation operations on groups should be done after each group is isolated in a single Dask DataFrame partition.
  • Persisting the data to disk is a useful way to avoid running the same operations multiple times and may be useful when debugging, as well.
  • Loading data from an SQL database in Dask, unlike in Pandas, requires SQLAlchemy.
  • In unit tests for data processing operations involving Dask DataFrames it is useful to check that the output is independent of the number of partitions.

Finally, as a general advice for a first project in Dask, to avoid unexpected behaviours, I recommend using a test-driven approach when engineering the data processing pipeline.
